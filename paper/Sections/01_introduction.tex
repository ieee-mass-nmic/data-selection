\section{Introduction}

Adapting large language models under limited compute requires efficiency on both the model side and the data side. A lab or platform rarely maintains a single adaptation setting: it keeps a \emph{registry} of parameter-efficient fine-tuning (PEFT) configurations---LoRA at several ranks and placements, adapters, (IA)$^3$, prefix tuning, or BitFit---chosen for different tasks, latency targets, memory limits, and deployment constraints. PEFT freezes the backbone and trains a small set of task-specific parameters, reducing optimizer memory and deployment storage. Data-efficient fine-tuning is the complementary tool: it selects a compact subset from a larger instruction pool, reducing the number of examples and training steps. These two tools are usually applied independently: PEFT controls which parameters move, while data selection controls which examples drive the update.

Current data selectors rarely model this interaction. Quality filters, diversity heuristics, proxy-model scores, and gradient-similarity methods often assign each example a single value for a target task and then reuse the selected subset across adaptation methods. That assumption is convenient, but PEFT methods change the trainable subspace itself. LoRA injects low-rank additive updates into selected projections~\cite{hu2022lora}; adapters insert residual bottlenecks~\cite{houlsby2019adapters}; and (IA)$^3$ learns multiplicative activation scalings~\cite{liu2022ia3}. An example whose gradient is aligned with a task inside one subspace need not be aligned inside another.

When the same pool must serve a whole registry, this omission becomes a cost problem. The strongest selectors are gradient- and influence-based: LESS~\cite{xia2024less} builds a target-specific gradient datastore and ranks candidates by their influence on a validation sketch. Because that datastore is tied to the target adapter's own trainable parameters, every new configuration pays to rebuild it, and selection cost grows linearly with the number of PEFTs served. The obvious shortcut---run the selector once and reuse its subset for every configuration---removes the cost but forfeits the quality that motivated a gradient selector: a subset chosen for one trainable subspace is mismatched to the others, and we measure that reusing a single LESS subset falls below even a PEFT-agnostic representation baseline (Table~\ref{tab:reuse-quality-cost}). This is the reuse-versus-recompute dilemma at the center of the paper: existing methods are either expensive per target or cheap but PEFT-agnostic in effect. The expensive supervision should instead be computed \emph{once} and reused across configurations.

Our motivation study shows that PEFT-specific utility changes with the trainable subspace. We estimate short-horizon utility for the same candidate pool under LoRA, adapter, and (IA)$^3$ variants, using two task sketches and independent anchor/seed replicates. Figure~\ref{fig:motivation-disagreement} shows that within-PEFT agreement exceeds cross-PEFT agreement. Same-family capacity changes retain moderate structure, but placement changes and cross-family changes sharply reduce rank agreement and Top-5\% overlap. A transfer matrix gives the downstream consequence: selecting data for the same PEFT used in training beats mismatched \emph{cross-family} selection by 2.0 points on average across target PEFTs (Appendix Table~\ref{tab:transfer}); the Analysis section reports the narrower 1.45-point within-LoRA configuration-scan gap.

This paper treats data value as a PEFT-conditioned quantity. We ask: given a candidate example $x$, a target task $t$, and a target PEFT configuration $p$, can we predict whether $x$ will help when the trainable parameters are exactly those exposed by $p$? This framing differs from PEFT-agnostic selection, which asks whether $x$ is generally good for the task. It also differs from recomputing an influence selector for every PEFT, because the expensive supervision should be reused across many target configurations.

PCU-Select (PEFT-Conditional Utility) addresses this question with an intervention-site utility layer. It maps samples, task sketches, and PEFT configurations into a shared set of 24 module-output sites in the backbone. Sample and task signatures describe gradient alignment at each site. The PEFT code describes which sites a method touches, how it acts there, and how much trainable capacity it has. PCU-Select uses a low-fidelity site-weighted gradient proxy for broad coverage, corrects it with short-horizon high-fidelity labels, and trains a conditional scorer that predicts both utility and uncertainty. At deployment, the scorer ranks the pool for a target $(p,t)$ and a quota allocator balances utility with semantic coverage.

We make three contributions.
\begin{itemize}
    \item \textbf{Problem.} We formulate cross-PEFT data-efficient fine-tuning as an \emph{amortization} problem: the goal is not to beat a per-PEFT influence selector on a single target but to preserve its quality while converting per-target recomputation into reusable supervision. We show empirically that data-value rankings differ across PEFT configurations (cross-family Top-5\% overlap near 0.17, a 2.0-point downstream cross-family mismatch penalty), and that reusing one influence-selected subset therefore drops below even a PEFT-agnostic baseline.
    \item \textbf{Method.} We introduce PCU-Select, a reusable PEFT-conditioned selector built from intervention-site gradient signatures, structured PEFT encodings, task sketches, multi-fidelity utility labels, uncertainty-aware scoring, and adaptive cluster quotas.
    \item \textbf{Evaluation.} Across GSM8K, HumanEval, MMLU, and TyDiQA on a fixed Llama-2-7B backbone family, PCU-Select improves 10\%-budget downstream quality over PEFT-agnostic selectors (over random by 2.89 points and RDS+ by 0.74 points) and near-ties a per-PEFT LESS reimplementation on the seen registry while still generating distinct per-target subsets. Its offline supervision amortizes after about 2.3 target configurations, reducing five-PEFT selection compute from 158.0 to 72.9 GPU-hours ($2.2\times$).
\end{itemize}
