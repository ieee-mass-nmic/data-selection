\section{Introduction}

Adapting large language models under limited compute requires efficiency on both the model and the data side. A lab or platform rarely maintains a single adaptation setting: it keeps a \emph{registry} of parameter-efficient fine-tuning (PEFT) configurations---LoRA at several ranks and placements, adapters, (IA)$^3$, prefix tuning, or BitFit---chosen for different tasks, latency targets, and memory limits. Data-efficient fine-tuning is the complementary tool: it selects a compact subset from a larger instruction pool, cutting examples and training steps. The two are usually applied independently: PEFT controls which parameters move, data selection controls which examples drive the update.

Current data selectors rarely model this interaction. Quality filters, diversity heuristics, proxy-model scores, and gradient-similarity methods often assign each example a single value for a target task and then reuse the selected subset across adaptation methods. That assumption is convenient, but PEFT methods change the trainable subspace itself: LoRA injects low-rank additive updates into selected projections~\cite{hu2022lora}, adapters insert residual bottlenecks~\cite{houlsby2019adapters}, and (IA)$^3$ learns multiplicative activation scalings~\cite{liu2022ia3}. An example aligned with a task inside one subspace need not be aligned inside another.

When one pool must serve a whole registry, this omission becomes a cost problem. The strongest selectors are gradient- and influence-based: LESS~\cite{xia2024less} builds a target-specific gradient datastore and ranks candidates by their influence on a validation sketch. Because that datastore is tied to the target adapter's trainable parameters, every configuration pays to rebuild it, and selection cost grows linearly in the number of PEFTs served. The obvious shortcut---select once and reuse the subset everywhere---removes the cost but forfeits the quality that motivated a gradient selector: a subset chosen for one subspace is mismatched to the others, and reusing a single LESS subset falls below even a PEFT-agnostic representation baseline (Table~\ref{tab:quality-cost-headline}). This is the reuse-versus-recompute dilemma at the center of the paper: existing methods are either expensive per target or cheap but PEFT-agnostic in effect.

Our motivation study shows that PEFT-specific utility changes with the trainable subspace. Figure~\ref{fig:motivation-disagreement} scores one candidate pool under LoRA, adapter, and (IA)$^3$ variants and shows that within-PEFT agreement exceeds cross-PEFT agreement: same-family capacity changes retain moderate structure, but placement and cross-family changes sharply reduce rank agreement and Top-5\% overlap. A transfer matrix gives the downstream consequence: selecting data for the same PEFT used in training beats mismatched \emph{cross-family} selection by 2.0 points on average across target PEFTs (Appendix Table~\ref{tab:transfer}); the Analysis section reports the narrower 1.45-point within-LoRA configuration-scan gap.

This paper treats data value as a PEFT-conditioned quantity. We ask: given a candidate example $x$, a target task $t$, and a target PEFT configuration $p$, can we predict whether $x$ will help when the trainable parameters are exactly those exposed by $p$? This framing differs from PEFT-agnostic selection, which asks whether $x$ is generally good for the task, and from per-PEFT influence recomputation, because the expensive supervision should be reused across many target configurations.

PCU-Select (PEFT-Conditional Utility) maps samples, task sketches, and PEFT configurations onto 24 shared module-output sites. Sample and task signatures capture site-level gradient alignment; a structured PEFT code encodes each method's sites, operator, and capacity. A low-fidelity gradient proxy provides broad coverage, short-horizon high-fidelity labels correct it, and a conditional scorer predicts utility and uncertainty. At deployment, the scorer ranks the pool for a target $(p,t)$ and a quota allocator balances utility with semantic coverage.

This design amortizes target-specific selection across a PEFT registry. Our contributions are three-fold:
\begin{itemize}
    \item \textbf{PEFT-conditioned utility.} We formulate data value as $u(x,p,t)$ and show that both data-value rankings and the downstream quality of selected subsets depend on the PEFT configuration (Figure~\ref{fig:motivation-disagreement}; Appendix Table~\ref{tab:transfer}).
    \item \textbf{Intervention-site amortization.} PCU-Select shares 24-site sample/task signatures and multi-fidelity supervision across configurations; a structured PEFT code retains target specificity without rebuilding a gradient datastore. Removing the PEFT code or task sketch costs 1.37 or 0.82 downstream points, respectively (Appendix Table~\ref{tab:ablations}).
    \item \textbf{Registry-scale quality and cost.} Across four tasks and five seen PEFTs on Llama-2-7B~\cite{touvron2023llama2}, PCU-Select gains 0.74 points over RDS+ and differs from per-PEFT LESS by $+0.04$. Its selection-stage cost per added four-task PEFT is $44\times$ lower (1.44 versus 63.2 GPU-hours; Tables~\ref{tab:main-results} and~\ref{tab:quality-cost-headline}).
\end{itemize}
